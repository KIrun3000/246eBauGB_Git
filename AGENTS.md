# Repository guidance

## Single source of truth

- The only authoritative working copy is on `mac-mini-von-loco` at `/Users/vkclaw/Documents/246eBauGB`.
- MacBook and iPhone sessions are control surfaces. Commands, edits, Git operations, checks and deployments must run against the Mac mini working copy.
- Before changing files, verify the repository root, `git remote -v`, `git status --short --branch` and `git worktree list`.
- If the active project is a MacBook clone or the path is not the Mac mini path above, do not edit it. Reconnect to the Mac mini project instead.
- Never let two tasks modify the same checkout or branch at the same time. Use a separate worktree and a separate `codex/<task>` branch for parallel work.

## User experience and autonomy

- The project owner is not expected to use a terminal or know Git, GitHub, CI or Vercel.
- Perform routine project work yourself. Do not ask the owner to run commands that Codex can safely run.
- A normal implementation task includes the complete delivery flow: inspect, create or reuse a task branch, implement, validate, review the diff, commit only intended files, push, create a pull request, monitor checks, merge when eligible and verify the resulting deployment.
- Do not pause for confirmation before routine branch creation, commits, pushes, pull requests, preview deployments or eligible merges when the requested change is already clear.
- Ask only when a choice materially changes the product, legal meaning, public availability, costs, credentials, DNS, account ownership or access rights.
- Explain results in plain German and lead with what is now done or what remains blocked.

## Git and worktree workflow

- Never work directly on `main`, push directly to `main`, force-push or rewrite shared history.
- Start each independent change from current `origin/main` on a short `codex/<task>` branch. Reuse an existing task branch only when it clearly belongs to the same unfinished task.
- Preserve unrelated or pre-existing changes. Stage explicit task files rather than using `git add -A` in a mixed worktree.
- Never use destructive cleanup such as `git reset --hard`, `git clean -fdx` or replacing the working directory.
- Use concise commits. Push the branch and create a ready pull request after all required checks pass locally.
- Repository automation may squash-merge low-risk `codex/*` pull requests after the required GitHub checks pass.
- Changes to legal content or automation infrastructure require their additional gate described below. Codex completes that gate; the owner should not have to operate GitHub manually.

## Required checks

- Use the existing Node 22 and npm runtime. Do not install global packages or change system, shell, Git, npm, SSH or Tailscale configuration.
- Install exactly from the lockfile with `npm ci`; do not update the lockfile unless the task intentionally changes dependencies.
- Before committing, run `npm run check` and `git diff --check`.
- Review the complete diff for secrets, accidental files, unsupported claims and unrelated edits.
- GitHub's `quality` check is required before merge. Do not bypass a failed or pending required check.

## Legal and factual publication gate

- Treat changes under `src/content/`, `research/`, MDX pages, legal/disclaimer pages, `CONTENT_DOD.md`, `PROMPT_CONTRACT.md` and project-specific legal skills as legally sensitive.
- Follow `CONTENT_DOD.md` and `PROMPT_CONTRACT.md`, but do not assume either file is current or error-free.
- Verify material legal claims against current official primary sources. Record the applicable version or date and link the exact source.
- Separate current law, administrative guidance, draft legislation, case law and inference. Never present a draft or unofficial interpretation as settled law.
- For legal-content pull requests, the `legal-review-required` label is added automatically. Do not merge until a second legal/factual review is documented in the pull request and the label has been replaced with `legal-reviewed`.
- NotebookLM may be used as an additional contradiction and coverage check after it is connected, but it is not the authoritative source.
- Never publish legal text solely because it was generated by an LLM or because the site builds successfully.

## Automation-infrastructure gate

- Changes to `.github/`, `.codex/`, `AGENTS.md`, `package.json`, lockfiles, `vercel.json`, deployment settings or security rules receive `infrastructure-review-required`.
- Review permissions, secret exposure, trigger safety and deployment impact before merging such changes. Replace the label with `infrastructure-reviewed` only after that review is complete.

## Deployment policy

- Pull-request branches may receive automatic Vercel preview deployments.
- Production may deploy only from protected `main` after the required checks and applicable legal or infrastructure gate pass.
- Never run a production deployment from a dirty checkout or a task branch.
- Automatic production deployment remains disabled until the known legal-content audit findings and required public legal details are resolved. Re-enabling it must be an intentional, reviewed infrastructure change.
- Do not change domains, DNS, billing, team membership, secrets or account-level Vercel settings without an explicit request.

## Local development and security

- Start the development server only on `127.0.0.1:4321`, never on a wildcard interface.
- Never commit secrets, credentials, private keys, `.env` files, `.vercel/` metadata, personal audit data or files copied from another clone without explicit approval.
- Untracked files from a MacBook clone are not automatically part of this repository.
